# AGENTS.md

## 1. 项目背景

本项目是一个从零开始开发的类 Netflix 视频网站。项目最终会包含影片浏览、搜索、详情、播放、收藏、观看历史、后台影片管理、视频上传、转码、评论、评分和弹幕等能力。

但是当前开发必须按阶段小步推进。用户是前端初学者，因此代码和文档都要适合初学者理解，不能一次性生成完整全栈项目。

当前阶段是第一阶段：前端 MVP 用户侧闭环。

## 2. 当前阶段目标

第一阶段目标是先完成视频网站前端用户侧最小闭环：

1. mock 数据中有影片信息。
2. 首页、分类页、搜索页可以展示和筛选影片数据。
3. 用户可以从影片卡片进入影片详情页。
4. 用户可以从详情页进入播放页。
5. 播放页使用 HTML5 `video` 播放测试视频地址。
6. 用户可以在本地体验观看历史、收藏、评分、评论和弹幕的最小版本。
7. 用户中心可以展示本地观看历史和本地收藏。

第一阶段只实现以下用户侧页面和本地交互：

- 首页
- 分类页
- 搜索页
- 影片详情页
- 播放页
- 用户中心
- 搜索和分类筛选
- localStorage 观看历史
- localStorage 收藏
- localStorage 评分
- localStorage 评论
- localStorage 弹幕

第一阶段暂时不要实现：

- 真实后端
- 数据库
- 视频上传
- 视频转码
- 评论、评分、弹幕的真实后端能力
- 真实登录
- 真实用户权限
- 支付
- 会员
- 推荐算法
- 复杂后台管理

## 3. 长期项目目标

项目最终目标是实现一个可逐步扩展的视频网站：

- 第一阶段：前端 MVP 用户侧闭环，使用 mock 数据和 `localStorage` 跑通浏览、搜索、筛选、详情、播放、观看历史、收藏、评分、评论、弹幕最小版本。
- 第二阶段：前端体验整理和接口接入准备，可继续优化页面细节、空状态、类型和代码组织，但不新增真实后端能力。
- 第三阶段：真实后端基础接口，包括用户、影片、分类、收藏、观看历史，以及简单影片管理。
- 第四阶段：后台 mp4 原文件上传和播放。
- 第五阶段：FFmpeg 转码、HLS、m3u8 播放和任务状态管理。
- 第六阶段：评论、评分和弹幕的真实后端接口、真实用户数据和后台审核管理。

任何开发任务都必须先确认当前阶段，不要提前实现后续阶段功能。

## 4. 技术栈约束

当前前端技术栈固定为：

- React
- TypeScript
- Vite

样式方案：

- 优先使用普通 CSS。
- 或使用 CSS Modules。

不要擅自引入：

- UI 组件库
- 状态管理库
- 后端框架
- 数据库
- 对象存储 SDK
- 上传组件库
- FFmpeg 或转码工具
- HLS 播放库
- 支付 SDK
- 复杂推荐算法相关依赖

如果确实需要新增依赖，必须先说明原因，并等待用户确认。

## 5. 开发方式约束

每次开发只实现一个小功能。

开始写代码前必须先阅读 `docs/` 目录下的需求文档，尤其是：

- `docs/01-product-prd.md`
- `docs/02-frontend-pages.md`
- `docs/06-development-plan.md`
- `docs/07-acceptance.md`

开发时必须遵守：

- 不要一次性重写整个项目。
- 不要擅自扩大需求范围。
- 不要为了“看起来完整”添加用户没有要求的功能。
- 不要提前实现后续阶段能力。
- 不要把 mock 阶段写成真实后端架构。
- 不要生成大量暂时用不到的文件。
- 不要做无关重构。

如果发现需求不清楚，先提出具体问题；如果可以合理假设，就选择最简单、最符合当前阶段的做法。

## 6. 目录结构约束

前端代码必须放在 `frontend/src` 下。

建议目录结构：

```text
frontend/
  src/
    components/
    pages/
    mocks/
    types/
    utils/
```

目录职责：

- `frontend/src/pages`：页面组件，例如首页、分类页、搜索页、详情页、播放页、用户中心。
- `frontend/src/components`：公共组件，例如导航栏、影片卡片、影片列表。
- `frontend/src/mocks`：mock 数据，例如 `videos.ts`。
- `frontend/src/types`：TypeScript 类型定义。
- `frontend/src/utils`：工具函数。

不要把所有代码都写在一个文件里。也不要为了追求架构感创建过多空目录或空文件。

## 7. 代码风格约束

代码要适合前端初学者阅读。

必须遵守：

- 组件职责清晰。
- 文件名有明确含义。
- 变量名有明确含义。
- TypeScript 类型要简单直接。
- 样式类名要能表达用途。
- 页面和组件之间的数据流要容易理解。

避免：

- 过度封装。
- 复杂抽象。
- 复杂泛型。
- 复杂状态管理。
- 无用代码。
- 废弃代码。
- 大段难懂的工具函数。
- 和当前阶段无关的配置。

可以添加少量注释，但注释必须解释关键意图，不要写无意义注释。

## 8. 第一阶段实现边界

第一阶段允许实现：

- React 页面路由。
- mock 影片数据。
- 影片卡片展示。
- 首页影片区域。
- 分类页基础筛选。
- 搜索页本地搜索。
- 影片详情页。
- HTML5 `video` 播放页。
- 用户中心展示本地观看历史和本地收藏。
- 基于 `localStorage` 的观看历史。
- 基于 `localStorage` 的收藏。
- 基于 `localStorage` 的评分。
- 基于 `localStorage` 的评论。
- 基于 `localStorage` 的弹幕。
- 基础响应式样式。

第一阶段不允许实现：

- 真实登录注册。
- 真实用户权限。
- 真实收藏接口。
- 真实观看历史接口。
- 真实评分接口。
- 真实评论接口。
- 真实弹幕接口。
- 真实搜索接口。
- 真实分类接口。
- 后台管理。
- 文件上传。
- 视频转码。
- HLS 播放。

第一阶段的收藏、观看历史、评分、评论和弹幕都只能是 `localStorage` 最小版本，不能伪装成真实登录用户、真实后端数据或后台审核能力。

## 9. 验证要求

每次修改后必须运行：

```bash
npm run build
```

如果项目在 `frontend` 目录下，则应该在 `frontend` 目录中运行构建命令。

构建失败时必须先修复构建错误，不要继续扩展新功能。

完成任务后的回复必须说明：

- 修改了哪些文件。
- 实现了什么功能。
- 是否运行了 `npm run build`。
- 构建是否通过。
- 有没有遗留问题。

如果因为项目尚未初始化等原因无法运行构建命令，必须明确说明原因。

- 真实视频上传。
- HLS 转码。
- FFmpeg 转码任务。
- 用户权限系统。
- 支付。
- 会员。
- 广告。
- CDN。
- 防盗链。
- DRM。
- 复杂推荐算法。
- 复杂后台管理系统。
- 评论、评分、弹幕的真实后端能力。
- 评论、评分、弹幕的后台审核管理。
- 多语言。
- App 或小程序。

禁止在没有确认的情况下添加新依赖。

禁止为了展示效果而添加与需求无关的功能。

## 11. 和文档保持一致

开发时必须以 `docs/` 目录中的阶段化文档为准。

如果代码实现和文档冲突，优先停下来说明冲突点，不要直接扩大范围。

如果用户要求进入下一阶段，先更新或确认相关文档，再开始实现。

## 12. 当前最重要的判断标准

当前阶段最重要的验收标准是：

用户能从首页看到 mock 影片，点击进入详情页，再点击进入播放页，并用 HTML5 `video` 播放测试视频。

在这条链路完成的基础上，第一阶段还允许使用 `localStorage` 做观看历史、收藏、评分、评论和弹幕的最小用户侧体验。不要扩展真实后台、上传、转码、数据库、真实登录或真实互动接口。
